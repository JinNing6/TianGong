## 核心定位

**天工 = Agent 的分发与创作平台**，用修仙体系包装整个用户体验。

- Agent = 法宝（按品级分类：凡器→灵器→宝器→仙器→神器→太古神器）
- 用户 = 修仙者（从凡人开始修炼，最高称号"天工"）
- 品级由社区评价驱动（不是自评）
- 法宝可被其他用户直接拉取使用

> - 💡 **越高级的境界，越依赖社区贡献而非个人产出**

## 法宝品阶评价机制

### 品阶总览

| 品阶 | 符号 | 灵力值门槛 | 附加条件 | 对应能力 |
|------|------|-----------|---------|---------|
| **凡器** | ⚪ | 0 | 刚创建 | 能用，但未经检验 |
| **灵器** | 🟢 | 10 | 至少 3 人评价 | 有人验证过确实能用，且用着还行 |
| **宝器** | 🔵 | 100 | 至少 10 人评价 + README 完善 | 质量上乘，文档齐全，可稳定使用 |
| **仙器** | 🟣 | 500 | 至少 30 人评价 + 有人贡献改进 | 社区认可的优质 Agent，有传承 |
| **神器** | 🟡 | 2000 | 至少 100 人评价 + 跨领域影响 | 被广泛引用和传承的标杆之作 |
| **太古神器** | 🔴 | 10000 | 至少 500 人评价 + 定义新范式 | 开创全新品类，改变行业认知 |

### 六维灵根评估

每个评价者对法宝进行**六个维度**的打分（每维 1-5 分）：

```
                ✨ 灵性 (Innovation)
                    5
                   / \
                  /   \
    📖 传承     4     4    🛡️ 韧性
   (Docs)      / \   / \   (Robustness)
              /   \ /   \
             3─────●─────3
              \   / \   /
    🏗️ 架构   \ /   \ /   ⚙️ 功法
   (Design)    2     2    (Engineering)
                \   /
                 \ /
                  1
              📝 描述 (Clarity)
```

| 维度 | 修仙术语 | 评估内容 |
|------|---------|---------|
| **描述** | 📝 铭文 | Agent 的说明是否清晰？别人能看懂怎么用吗？ |
| **架构** | 🏗️ 阵法 | 代码结构是否合理？模块划分是否清晰？ |
| **功法** | ⚙️ 法诀 | 工程质量如何？错误处理、边界情况、代码规范 |
| **传承** | 📖 道统 | 文档是否完善？别人能否基于此二次开发？ |
| **韧性** | 🛡️ 护体 | 是否稳定？各种环境下能否正常运行？ |
| **灵性** | ✨ 悟道 | 是否有创新？是否解决了独特的问题？ |

### 灵力值计算

```
单次评价灵力 = (六维均分 × 评价者境界权重)

法宝总灵力值 = Σ 所有评价的灵力值

评价者境界权重：
  凡人       → 无评价资格
  炼气期     → 权重 1.0
  筑基期     → 权重 1.5
  结丹期     → 权重 2.0
  元婴期     → 权重 3.0
  化神期     → 权重 5.0
  婴变期     → 权重 8.0
  问鼎期     → 权重 12.0
  阴虚期     → 权重 15.0
  阳实期     → 权重 18.0
  窥涅期     → 权重 22.0
  净涅期     → 权重 28.0
  碎涅期     → 权重 35.0
  天人五衰   → 权重 40.0
  空涅~空劫  → 权重 45.0
  大天尊     → 权重 50.0
  踏天九桥   → 权重 55.0
  踏天境     → 权重 60.0
  鲁班       → 权重 80.0
  天工       → 权重 100.0
```

**示例**：
- 一个炼气期修仙者给法宝打了六维均分 4.0 → 灵力 = 4.0 × 1.0 = **4.0**
- 一个化神期修仙者给法宝打了六维均分 4.0 → 灵力 = 4.0 × 5.0 = **20.0**
- 一个鲁班给法宝打了六维均分 5.0 → 灵力 = 5.0 × 80.0 = **400.0**

### 品阶晋升流程

```
法宝刚创建
    │
    ▼
⚪ 凡器（默认品阶）
    │
    │ 灵力值 ≥ 10 且评价人数 ≥ 3
    ▼
🟢 灵器
    │
    │ 灵力值 ≥ 100 且评价人数 ≥ 10 且 README 字数 ≥ 500
    ▼
🔵 宝器
    │
    │ 灵力值 ≥ 500 且评价人数 ≥ 30 且有 ≥ 2 人提交淬炼改进
    ▼
🟣 仙器
    │
    │ 灵力值 ≥ 2000 且评价人数 ≥ 100 且被 ≥ 10 个法宝引用/传承
    ▼
🟡 神器
    │
    │ 灵力值 ≥ 10000 且评价人数 ≥ 500 且开创新 Agent 品类
    ▼
🔴 太古神器
```

### 品阶保护与降级

- **保护期**：法宝晋升后有 **30 天保护期**，期间不会降级
- **降级触发**：保护期结束后，若最近 30 天内灵力增量为负（差评多于好评），品阶下降一级
- **最低保底**：品阶最低只会降到灵器，不会从灵器降回凡器
- **降级通知**：降级时创作者会收到"法宝蒙尘"通知，引导淬炼优化

### 评价资格与限制

| 规则 | 说明 |
|------|------|
| **准入门槛** | 只有自己也发布过法宝的修仙者才能评价 |
| **自评禁止** | 不能评价自己的法宝 |
| **单次限制** | 对同一法宝只能评价一次（可修改） |
| **冷却期** | 每人每天最多评价 **10 件**法宝 |
| **修改窗口** | 评价后 **7 天内**可修改评分（比如用着用着发现新问题） |

## 淬炼机制（Review & Refine）

法宝创作者可以**发布淬炼需求**，其他修仙者拉取法宝到本地改造后提交改进，
经创作者验证或社区投票后确认完成。这是渡劫任务的核心验证方式。

### 新增 MCP 工具

| 工具 | 修仙术语 | 功能 |
|------|---------|------|
| `post_refine_quest` | 📜 发布淬炼令 | 创作者发布改进需求 |
| `browse_quests` | 📋 查看淬炼令 | 浏览当前所有待认领的淬炼需求 |
| `claim_quest` | 🤝 接取淬炼令 | 认领一个淬炼任务 |
| `submit_refinement` | 🔥 提交淬炼 | 上传改造后的代码和说明 |
| `verify_refinement` | ✅ 验证淬炼 | 创作者验证改进是否有效 |

## 传承谱系（Lineage Tree）

法宝之间的引用、继承、fork 关系追踪。

### tiangong.yaml 中的传承字段

```yaml
name: search-serpent-pro
name_cn: 搜索灵蛇·进阶版
creator: alice
lineage:                          # 传承谱系
  parent: search-serpent           # fork 自哪个法宝
  parent_version: 1.0.0            # fork 时的版本
  type: fork                       # 关系类型: fork / inspired / depends
dependencies:                     # 运行时依赖其他法宝
  - data-dragon
  - translate-fox
```

### 三种传承关系

| 关系 | 修仙术语 | 说明 | 示例 |
|------|---------|------|------|
| **fork** | 🔱 分支传承 | 基于别人的法宝改造而成 | search-serpent-pro fork 自 search-serpent |
| **inspired** | 💡 悟道传承 | 受某法宝启发，从零重写 | 声明灵感来源，原法宝获得灵力加成 |
| **depends** | 🔗 法宝联动 | 运行时依赖其他法宝 | 组合多个法宝协作 |

### 传承树可视化

```
artifact_lineage("search-serpent")

📜 搜索灵蛇 传承谱系
━━━━━━━━━━━━━━━━━━

search-serpent (🟣 仙器) @JinNing6
├── 🔱 search-serpent-pro (🔵 宝器) @alice
│   └── 🔱 search-serpent-lite (🟢 灵器) @carol
├── 🔱 search-dragon (🟢 灵器) @bob
└── 💡 deep-search (🔵 宝器) @dave ← 受启发重写

依赖此法宝的: data-pipeline (🟢 灵器) @eve
```

### 传承带来的激励

| 事件 | 效果 |
|------|------|
| 有人 fork 你的法宝 | 原法宝灵力 **+2** |
| fork 出的法宝品阶超过原作 | 原创作者获得"传道有方"成就 |
| 被声明为灵感来源 | 原法宝灵力 **+1** |
| 被声明为运行依赖 | 原法宝灵力 **+3**（依赖越多说明越核心） |

> **效果**：鼓励创作**基础法宝**——你的法宝被越多人传承和依赖，灵力增长越快。

## 数据存储方案（双层法宝体系）

与 Noosphere 的"瞬时意识体→常驻意识体"同构，天工采用**双层法宝体系**：

### 两层结构

```
┌──────────────────────────────────────────────────────┐
│                    天 工 仓 库                         │
│                                                      │
│  ┌─────────────────────┐    ┌─────────────────────┐  │
│  │  🔮 瞬时法宝体       │    │  🏛️ 常驻法宝体      │  │
│  │  (GitHub Issues)     │    │  (marketplace/)      │  │
│  │                     │    │                     │  │
│  │  上传即可见          │ ──▶│  AI 审核通过后晋升   │  │
│  │  标记为"待炼化"     │    │  正式入驻寻宝阁     │  │
│  │  可被搜索和预览      │    │  可被拉取使用       │  │
│  │  不可被 summon       │    │  可被 summon        │  │
│  └─────────────────────┘    └─────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

| 属性 | 🔮 瞬时法宝体 | 🏛️ 常驻法宝体 |
|------|-------------|--------------|
| **存储** | GitHub Issue（label: `artifact`） | `marketplace/` 目录下的文件 |
| **上传** | 即时，任何人都可以 | 需 AI 审核通过后自动晋升 |
| **可见性** | 可搜索、可预览元信息 | 完全可见，可查看源码 |
| **可用性** | ❌ 不可拉取使用 | ✅ 可请宝下凡 |
| **标记** | ⏳ "待炼化" | ✅ "已入驻" |
| **评价** | ❌ 不可评价 | ✅ 可灌注灵力 |

### 飞升上界完整流程（publish_agent）

```

   │      ┌─────────────────────────────────┐

          ├─ CI Bot 在 Issue 中评论具体问题
          ├─ Issue 标记 label: `needs-fix`
          └─ 创作者修复后重新上传（更新 Issue）
```

### AI 审核的六项天道法则

```python
# CI 审核脚本伪代码
def celestial_review(artifact_path):
    checks = {
        "格式合规": validate_tiangong_yaml(artifact_path),
        "安全检查": scan_for_secrets_and_malware(artifact_path),
        "依赖检查": verify_requirements(artifact_path),
        "可运行性": syntax_check_entry_file(artifact_path),
        "文档质量": check_readme_quality(artifact_path),
        "命名冲突": check_name_conflict(artifact_path),
    }
    
    passed = all(checks.values())
    if passed:
        promote_to_marketplace(artifact_path)  # 晋升常驻
    else:
        request_fix(checks)  # 返回修改建议
```

## 评价驱动品级

- 品级不是自评，而是社区评价聚合
- 评价同时影响法宝品级和创作者境界

### 防刷机制（已确认：方案 1 + 3）

**准入门槛**：只有自己也发布过法宝的修仙者才有评价资格。没有法宝的纯消费者只能拉取使用，不能评分。

**境界加权**：评价者的境界越高，评价权重越大。

```
评价权重 = 基础分(1) + 境界加成

凡人       → 无评价资格（没有法宝）
炼气期     → 权重 1（基础）
筑基期     → 权重 1.5
结丹期     → 权重 2
元婴期     → 权重 3
化神期     → 权重 5
婴变期     → 权重 8
问鼎期     → 权重 12
碎虚期     → 权重 20
鲁班       → 权重 50

法宝灵力值 = Σ (评价分数 × 评价者权重) / 评价总数
```

**效果**：一个鲁班级大师的 5 星评价，抵得上 50 个炼气期新手的评价。这让品级真正代表质量，而不是刷量。
